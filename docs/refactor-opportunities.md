## 1. Repository Overview

`blackjack-trainer` is now a **two-implementation** project for practicing blackjack
basic strategy, Hi-Lo card counting (running + true count), deviations, and a
post-count showdown:

```
src/            Angular 21 web app — the source of truth for game logic
  app/core/models       pure types + pure rules (card, hand, shoe, showdown, …)
  app/core/services     engines (basic-strategy, deviation, counting, showdown) + stats stores
  app/data              static charts (S17/H17 strategy + deviations) and counting systems
  app/shared            reusable presentational components
  app/features          one folder per route (basic-strategy, card-counting, deviations)
ios/            Native SwiftUI app — re-implements the same engines
  BlackjackTrainer/Engine     Swift ports of the TS engines/models
  BlackjackTrainer/Views      SwiftUI screens + components
  Fixtures/                   machine-written parity vectors (do not hand-edit)
tools/          export-parity-fixtures.ts — emits ios/Fixtures from the TS engines
```

**Parity contract (important):** the TypeScript engines are the single source of
truth. `tools/export-parity-fixtures.ts` serializes their charts and exhaustive
input→output "golden vectors" into `ios/Fixtures/*.json`; the Swift test target
asserts its ports reproduce every row. CI regenerates the fixtures and fails on any
diff (anti-drift gate). **Any change to TS engine/chart/counting logic must be
followed by `npm run export:fixtures` with no resulting `ios/Fixtures` diff.**

Boundaries are clean and conventional in both trees. The maintainer has been actively
refactoring toward shared components and single-source-of-truth helpers (recent TS
runs hoisted `TEN_VALUE_RANKS`, `softNonAceValue`, and the shared `Scenario` type into
`card.model.ts`).

## 3. Highest-Value Refactor Opportunities

### Opportunity F — Restore Prettier-cleanliness of the two tracked iOS-roadmap docs

- **Location:** `docs/ios-app-roadmap.md`, `docs/ios-app-roadmap-progress.md`.
- **Problem:** Both files violate the repo's Prettier config (a wrapped inline-code
  span's continuation indentation in the roadmap; Markdown table column padding in the
  progress log). `prettier --check .` flags them, so `npm run lint` and the CI
  `validate` job fail on `main`.
- **Why it matters:** A red lint gate masks future formatting regressions and blocks
  the project's own quality bar. These are docs — there is no behavior to preserve.
- **Suggested refactor:** Run the repo's own formatter on exactly these two files
  (`npx prettier --write` on each). The diff is mechanical (2 lines + 23 lines) and
  matches what `npm run format` produces.
- **Risk level:** Low
- **Expected benefit:** `npm run lint` / CI `validate` green again.
- **Suggested validation:** `npm run lint` (must exit 0).
- **Dependency ordering:** Independent. Do first — restores the gate the later
  cleanup is validated against.
- **Autopilot status:** Implemented

### Opportunity G — Dedupe the repeated insurance-explanation literal

- **Location:** `src/app/core/services/deviation-evaluator.service.ts`
  (`DeviationEvaluatorService.evaluate`, the insurance branch).
- **Problem:** The correct/incorrect ternary builds the explanation from the **same
  ~90-character template literal** twice; the incorrect branch only appends
  `You picked <action>.` The base sentence is duplicated verbatim, so the two copies
  can drift.
- **Why it matters:** One sentence, one source. The relationship ("incorrect = base +
  suffix") becomes explicit instead of two parallel literals.
- **Suggested refactor:** Bind the base sentence to a local `const`, then return it as
  the correct-branch explanation and `` `${base} You picked …` `` for the incorrect
  branch. Byte-identical output.
- **Risk level:** Low
- **Expected benefit:** −1 duplicated literal; clearer intent. The spec only asserts
  `toContain('Take insurance')`, so the change is contract-safe.
- **Suggested validation:** `CI=true npm test`, `npm run build`, `npm run lint`,
  `npm run export:fixtures` + `git diff --exit-code -- ios/Fixtures` (expected: no
  fixture change — the evaluator is not part of the exporter's path).
- **Dependency ordering:** Independent.
- **Autopilot status:** Planned

## 5. Larger Refactors (deferred — need human judgment)

- **Shared localStorage-backed JSON store.** `StatsStore`
  (`stats-store.ts`) and `ShowdownStatsService` (`showdown-stats.service.ts`) share
  the same load/persist **mechanics** (`typeof localStorage` guard → `JSON.parse` in
  try/catch → per-field numeric validation → tolerant `setItem`). The duplication is
  shallow and the two hold **intentionally different shapes** (correct/incorrect vs
  win/lose/push). Extracting a generic `readJson`/`writeJson` helper is plausible but
  requires a design call (how the per-field validator is expressed) and touches a
  persistence path — **left to human judgment.**
- **Shared trainer keydown handler.** Four components
  (`basic-strategy-page`, `deviations-page`, `card-counting-page`, `showdown`) carry
  `@HostListener('window:keydown')` handlers with the same Enter→next / key→answer
  skeleton but different gating (e.g. the Deviations page's `canDealNextHand()`). A
  shared helper in `core/keyboard.ts` could absorb the skeleton, but the variations
  are **behavioral** — needs judgment. **Not auto-implemented.**
- **Unified signed-integer formatter.** `formatTrueCount` and the count-feedback
  `deltaLabel` share the "+N for positives" shape across **different domains** (true
  count vs per-card value). Merging risks over-abstraction. **Judgment call.**

## 6. Things Not Worth Refactoring Yet

- **The `StatsStore` subclasses** (basic-strategy / card-counting / true-count /
  deviation / deck-estimation) are an intentional Angular DI pattern: each is a
  distinct injectable singleton bound to its own `localStorage` key. `DeckEstimation`
  and the others already extend `StatsStore`; only `ShowdownStatsService` differs
  (different shape). Leave as-is.
- **`labelFor(a) => ACTION_LABELS[a]` template wrappers** — idiomatic, trivial.
- **`cleanupLegacyStatsKeys`** — a documented, wired-up (`main.ts`), tested
  compatibility shim. Not dead code.
- **`isValidIntegerAnswer` / `isValidDecimalAnswer` / `parseManualTrueCount`** —
  similar integer/decimal regexes but different return types and range semantics; not
  true duplication.
