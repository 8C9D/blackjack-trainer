# Refactor Opportunities Report

_Generated by the Repo Cleanup Autopilot on 2026-05-28. Branch: `chore/repo-cleanup-autopilot`._

## 1. Repository Overview

`blackjack-trainer` is a frontend-only **Angular 21** application (standalone
components, signals, lazy-loaded routes) for practicing blackjack basic
strategy, Hi-Lo card counting (running + true count), and Hi-Lo deviations.

```
src/app/
  app.{ts,config,routes}      app shell, DI config, lazy routes
  core/
    keyboard.ts               action hotkey bindings + key-guard helpers
    models/                   pure type/model modules (card, strategy, deviation, …)
    services/                 engines (basic-strategy, deviation, counting) + stats stores
  data/                       static charts (S17/H17 basic strategy + deviations), Hi-Lo system
  shared/                     reusable presentational components (card-image, stats-panel,
                              feedback-shell, rule-controls)
  features/
    basic-strategy/           page + action-buttons, blackjack-table, feedback-panel
    card-counting/            page + settings, card-stream, answer-form, feedback-panel
    deviations/               page + settings, feedback-panel, scenario-generators
```

Boundaries are clean and conventional: `core/models` are pure types, `core/services`
hold logic (engines are pure/deterministic with an injectable random seam), `data`
is static chart data, `shared` is cross-feature UI, and each `features/*` folder owns
one route. Recent git history shows the maintainer is already actively refactoring
toward shared components (`extract shared feedback shell`, `extract shared rule
controls`, `extract deviation evaluator service`, `lazy load feature routes`).

## 3. Highest-Value Refactor Opportunities

### Opportunity A — Remove duplicated `formatTrueCount` in the deviation feedback panel

- **Location:** `src/app/features/deviations/deviation-feedback-panel.component.ts`
  (private `formatTrueCount`) vs `src/app/core/services/deviation-evaluator.service.ts`
  (exported `formatTrueCount`).
- **Problem:** The component re-implements the exact same formatter
  (`tc > 0 ? `+${tc}` : String(tc)`) that is already exported and unit-tested in the
  evaluator service.
- **Why it matters:** Two copies of the same true-count display rule can drift
  (e.g. one starts showing `±0`); one is tested, the other is not.
- **Suggested refactor:** Import the exported `formatTrueCount` and reference it as a
  protected field so the template and `formatThreshold` reuse it. The page component
  already imports `formatTrueCount` from the same module, so the dependency direction
  is established.
- **Risk level:** Low
- **Expected benefit:** Single source of truth for true-count display; -1 duplicate.
- **Suggested validation:** `npm test` (deviations-page specs render the panel), `npm run build`.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented

### Opportunity B — De-duplicate the ten-value rank literal in scenario generators

- **Location:** `src/app/features/deviations/scenario-generators.ts`
  (`cardOfValue` and `tenValueCard`).
- **Problem:** The literal `['10', 'J', 'Q', 'K']` is written twice, and
  `cardOfValue(10, …)` duplicates the body of `tenValueCard`.
- **Why it matters:** Two definitions of "the ten-value ranks" can diverge; the
  overlap is needless.
- **Suggested refactor:** Extract a module-level `TEN_VALUE_RANKS` const and have
  `cardOfValue`'s `value === 10` branch delegate to `tenValueCard`. The change
  preserves the exact `random()` call order/count (rank index, then suit), so the
  seeded generator tests stay deterministic.
- **Risk level:** Low
- **Expected benefit:** One definition of the ten-value ranks; less branching.
- **Suggested validation:** `npm test` (scenario-generators specs use seeded RNG),
  `npm run build`.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented

### Opportunity C — Collapse `deviationRulesFor` into the core `deviationsFor`

- **Location:** `src/app/features/deviations/scenario-generators.ts`
  (`deviationRulesFor`) vs `src/app/core/services/deviation-engine.service.ts`
  (`deviationsFor`).
- **Problem:** Two exported functions with byte-identical bodies
  (`ruleSet === 'H17' ? H17_DEVIATIONS : S17_DEVIATIONS`).
- **Why it matters:** Both pick the deviation table for a rule set; having two
  copies invites drift if a third rule set is ever added.
- **Suggested refactor:** Have `deviationRulesFor` delegate to the core
  `deviationsFor` (feature → core, the allowed direction — the spec file already
  imports `deviationsFor` from core). Drop the now-unused `H17_DEVIATIONS` /
  `S17_DEVIATIONS` imports from `scenario-generators.ts`. The public symbol
  `deviationRulesFor` and its signature are preserved, so its dedicated test stays
  green.
- **Risk level:** Low–Medium (adds a feature→core import; behavior identical).
- **Expected benefit:** Single source of truth for rule-set table selection.
- **Suggested validation:** `npm test` (both functions and their consumers are
  tested), `npm run build`.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented

## 5. Larger Refactors

- **Shared keyboard handler for the two trainer pages.**
  `basic-strategy-page.component.ts` and `deviations-page.component.ts` have nearly
  identical `@HostListener('window:keydown')` handlers (Enter → next hand, otherwise
  `actionForKey` → answer). They differ only in the Deviations page's extra
  `canDealNextHand()` gate. A shared helper in `core/keyboard.ts` taking callbacks
  could remove the duplication, but it is **behavioral** and the two are not exactly
  the same, so it needs human judgment. **Not auto-implemented.**

- **Unified signed-integer formatter.** `formatTrueCount` and the
  `count-feedback-panel` breakdown `deltaLabel` (`delta > 0 ? `+${delta}` : …`) share
  the "+N for positives" shape but are different domains (true count vs per-card
  count value). Merging into one `formatSigned` util is possible but risks
  over-abstracting two semantically distinct labels. **Left as a judgment call.**

## 6. Things Not Worth Refactoring Yet

- **The four `StatsStore` subclasses** (`BasicStrategyStatsService`,
  `CardCountingStatsService`, `TrueCountStatsService`, `DeviationStatsService`) look
  like duplication but are an intentional Angular DI pattern: each is a distinct
  injectable singleton bound to its own `localStorage` key. Collapsing them would
  lose the per-feature injection tokens. Leave as-is.
- **`labelFor(a) => ACTION_LABELS[a]` wrappers** in several components — these exist
  only so templates can index the label map; idiomatic and trivial.
- **`cleanupLegacyStatsKeys`** — a documented, wired-up (`main.ts`), and tested
  compatibility shim. Not dead code; do not remove.
- **`isValidIntegerAnswer` vs `parseManualTrueCount`** — similar integer-regex
  validation but different return types and range semantics; not true duplication.
