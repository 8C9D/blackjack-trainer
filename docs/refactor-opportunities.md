## 1. Repository Overview

`blackjack-trainer` is a frontend-only **Angular 21** application (standalone
components, signals, lazy-loaded routes) for practicing blackjack basic
strategy, Hi-Lo card counting (running + true count), and Hi-Lo deviations.

```
src/app/
  app.{ts,config,routes}      app shell, DI config, lazy routes
  core/
    keyboard.ts               action hotkey bindings + key-guard helpers
    models/                   pure type/model modules (card, strategy, deviation, …)
    services/                 engines (basic-strategy, deviation, counting) + stats stores
  data/                       static charts (S17/H17 basic strategy + deviations), Hi-Lo system
  shared/                     reusable presentational components (card-image, stats-panel,
                              feedback-shell, rule-controls)
  features/
    basic-strategy/           page + action-buttons, blackjack-table, feedback-panel
    card-counting/            page + settings, card-stream, answer-form, feedback-panel
    deviations/               page + settings, feedback-panel, scenario-generators
```

Boundaries are clean and conventional: `core/models` are pure types, `core/services`
hold logic (engines are pure/deterministic with an injectable random seam), `data`
is static chart data, `shared` is cross-feature UI, and each `features/*` folder owns
one route. The maintainer has been actively refactoring toward shared components
and a single source of truth (recent: hoisting `TEN_VALUE_RANKS` to the card model,
extracting the deviation evaluator service, lazy-loading routes).

## 3. Highest-Value Refactor Opportunities

### Opportunity D — Replace the local `GeneratedScenario` with the shared `Scenario` type

- **Location:** `src/app/features/deviations/scenario-generators.ts`
  (`GeneratedScenario`) vs `src/app/core/models/card.model.ts` (`Scenario`).
- **Problem:** `GeneratedScenario` is a byte-for-byte copy of the shared `Scenario`
  interface (`readonly player: readonly [Card, Card]; readonly dealerUpcard: Card`).
  `Scenario` already exists as the canonical "initial two-card deal" type — the card
  generator returns it and `DeviationScenario extends Scenario`.
- **Why it matters:** Two definitions of the same scenario shape can drift; the local
  copy duplicates a type the model already owns. The exported `GeneratedScenario` is
  referenced **only inside its own file** (its declaration + the return type of
  `generateScenarioForDeviationRule`); the deviations page destructures
  `{ player, dealerUpcard }` and never names the type.
- **Suggested refactor:** Import `Scenario` from the card model (the file already
  imports several symbols from it), make `generateScenarioForDeviationRule` return
  `Scenario`, and delete the `GeneratedScenario` interface. Structural typing keeps
  every call site compatible.
- **Risk level:** Low
- **Expected benefit:** Single source of truth for the scenario shape; −1 duplicate
  type.
- **Suggested validation:** `npm test` (scenario-generators + deviations-page specs),
  `npm run build` (template type-check).
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented

### Opportunity E — Extract a `softNonAceValue` helper for the soft-hand decomposition

- **Location:** `src/app/core/services/basic-strategy-engine.service.ts`
  (`resolveSoft`) and `src/app/core/services/deviation-engine.service.ts`
  (`classifyForDeviation`).
- **Problem:** Both engines decompose a soft two-card hand the same way:
  `const nonAce = isAce(player[0]) ? player[1] : player[0];` followed by
  `cardHighValue(nonAce)` to get the non-ace value (2..10). The two lines are
  identical in both files.
- **Why it matters:** This is the soft-hand domain rule ("which card is the
  non-ace, and what is it worth"). Two independent copies can drift if the hand
  representation ever changes; both engines should derive it from one helper.
- **Suggested refactor:** Add a pure `softNonAceValue(player)` helper to
  `card.model.ts` (where `isAce`/`cardHighValue` already live and both engines
  already import from). Replace both call sites with it. Behavior is identical.
- **Risk level:** Low
- **Expected benefit:** One definition of the soft-hand non-ace value; both engines
  share it.
- **Suggested validation:** `npm test` (basic-strategy-engine, deviation-engine, and
  classifyForDeviation specs all exercise soft hands), `npm run build`.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented

## 5. Larger Refactors (deferred — need human judgment)

- **Shared keyboard handler for the two trainer pages.**
  `basic-strategy-page.component.ts` and `deviations-page.component.ts` have nearly
  identical `@HostListener('window:keydown')` handlers (Enter → next hand, otherwise
  `actionForKey` → answer). They differ in the Deviations page's extra
  `canDealNextHand()` gate on both the Enter and next-hand paths. A shared helper in
  `core/keyboard.ts` taking callbacks could remove the duplication, but it is
  **behavioral** and the two are not identical, so it needs human judgment. **Not
  auto-implemented.**

- **Unified signed-integer formatter.** `formatTrueCount`
  (`tc > 0 ? `+${tc}` : String(tc)`) and the `count-feedback-panel` breakdown
  `deltaLabel` (`delta > 0 ? `+${delta}` : …`) share the "+N for positives" shape but
  are different domains (true count vs per-card count value). Merging into one
  `formatSigned` util risks over-abstracting two semantically distinct labels.
  **Left as a judgment call.**

## 6. Things Not Worth Refactoring Yet

- **The four `StatsStore` subclasses** (`BasicStrategyStatsService`,
  `CardCountingStatsService`, `TrueCountStatsService`, `DeviationStatsService`) look
  like duplication but are an intentional Angular DI pattern: each is a distinct
  injectable singleton bound to its own `localStorage` key. Collapsing them would
  lose the per-feature injection tokens. Leave as-is.
- **`labelFor(a) => ACTION_LABELS[a]` wrappers** in several components — these exist
  only so templates can index the label map; idiomatic and trivial.
- **`cleanupLegacyStatsKeys`** — a documented, wired-up (`main.ts`), and tested
  compatibility shim. Not dead code; do not remove.
- **`isValidIntegerAnswer` vs `parseManualTrueCount`** — similar integer-regex
  validation but different return types (`boolean` vs `number | null`) and range
  semantics; not true duplication.
