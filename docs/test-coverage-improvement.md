## 3. Highest-Value Coverage Gaps

### Gap 1 — `shuffle` produces an invalid permutation (LIKELY BUG) ⚠️
- **Location**: `src/app/core/deck.ts:19-29`
- **Why it matters**: A shuffle must be a permutation (same multiset of cards out as in). The swap is malformed:
  ```ts
  const tmp = result[i];
  result[i] = result[j];
  result[i] = tmp;   // <-- overwrites the line above; result[i] reverts
  result[j] = tmp;
  ```
  Net effect each iteration: `result[i]` is unchanged and `result[j]` is overwritten with `result[i]`'s value — elements get **duplicated and lost**. The output is not a valid deck.
- **Existing tests**: `deck.spec.ts` checks only `length` and "does not mutate original" — both pass despite the bug.
- **Missing cases**: shuffle preserves the multiset of cards (sorted output equals sorted input); every card appears exactly once.
- **Suggested test**: assert `shuffle(deck, rng)` sorted-equals `deck` sorted (a true permutation check).
- **Status**: **Skipped** — a correct test fails against current code; fixing requires a production change (`result[j] = tmp` only). Mitigating factor: `shuffle` currently has **no production caller** (referenced only by its spec), so the defect is dormant. Recommend a human fix the swap and then add the permutation test.
- **Risk level (of a fix)**: Low. **Risk of test alone**: would break the build.
- **Suggested validation**: `npm run test:ci`.

### Gap 2 — `handTotal` ace adjustment never decrements ace count (LIKELY BUG) ⚠️
- **Location**: `src/app/core/scenario.ts:46-61`
- **Why it matters**: The soft-ace reduction loop is:
  ```ts
  while (total > 21 && aces > 0) {
    total -= 10;
    aces++;
    aces--;   // net no-op: aces never decreases
  }
  ```
  Because `aces` never drops, the loop keeps subtracting 10 until `total <= 21` even when no aces remain to demote. For hands that should still bust with all aces counted as 1, it under-reports the total. Example: `A,10,10,10` returns **21** but the true total is **31** (bust).
- **Existing tests**: `scenario.spec.ts` covers only `A,K` (blackjack = 21), which is correct even with the bug.
- **Missing cases**: multi-card hands where all aces must count as 1 and the hand still busts (`A,10,10,10` → 31); confirm correct soft totals (`A,A` → 12, `A,6` → 17).
- **Status**: **Skipped** for the busting case — a correct assertion (`expect(handTotal([A,10,10,10])).toBe(31)`) fails against current code. Mitigating factor: in-app, `handTotal` only ever receives **2-card** hands (via `generateStrategyScenario`), so the bug is latent. The **safe** soft-total cases (`A,A`→12, `A,6`→17) are correct today and are implemented (see §5, improvement 3 candidate / actually covered separately). Recommend a human change `aces++; aces--;` → `aces--;`, then add the busting-hand regression test.
- **Risk level (of a fix)**: Low. **Risk of busting-case test alone**: would break the build.
- **Suggested validation**: `npm run test:ci`.

### Gap 3 — `trueCount` guard clauses and half-deck rounding untested (HIGH VALUE, SAFE) ✅
- **Location**: `src/app/core/counting.ts:35-44`
- **Why it matters**: True count drives every counting/insurance decision in the app. Three behaviors are uncovered: (a) `decksRemaining <= 0` returns the raw running count; (b) `0 < decksRemaining < 0.25` rounds to 0 and also returns running; (c) decks-remaining is rounded to the nearest **half-deck** before dividing.
- **Existing tests**: a single case `trueCount(6, 2) === 3`.
- **Missing cases**: zero/negative decks-remaining guard; the rounded-to-zero guard; rounding up/down to nearest 0.5.
- **Status**: **Implemented** (§5).
- **Risk level**: Low. **Validation**: `npm run test:ci`.

### Gap 4 — `insuranceDecision` positive path and full shape untested (HIGH VALUE, SAFE) ✅
- **Location**: `src/app/core/insurance.ts:18-30`
- **Why it matters**: The "take insurance" branch is the whole point of the module, but it is only ever exercised via the trivial `shouldTakeInsurance(3)`. The end-to-end path (cards → running → true count → decision) is never tested as `shouldTake = true`, and the non-Ace short-circuit only asserts `offered`, not the full returned object.
- **Existing tests**: threshold true/false on `shouldTakeInsurance`; `insuranceDecision` offered/not-offered (`offered` field only).
- **Missing cases**: Ace + high running count → `shouldTake true`, `offered true`, `trueCount ≥ 3`; Ace + zero count → `shouldTake false`, `trueCount 0`; non-Ace returns the exact `{ offered:false, shouldTake:false, trueCount:0 }` regardless of cards.
- **Status**: **Implemented** (§5).
- **Risk level**: Low. **Validation**: `npm run test:ci`.
