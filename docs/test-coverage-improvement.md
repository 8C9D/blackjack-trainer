## 2. Current Coverage Quality Summary

The decision-critical engines (`basic-strategy-engine`, `counting-engine`,
`deviation-engine`, `deviation-evaluator`) and the `stats-store` base all
have specs, and several card-counting / deviations components are tested.
The most notable **untested** areas are:

1. **`card-generator.service.ts`** — generates the random cards/sequences
   that drive every drill, yet has no spec. Generation logic is the kind of
   code that benefits most from seeded/deterministic tests (length, value
   ranges, distribution, edge counts).
2. **The four `StatsStore` subclasses** — `basic-strategy-stats`,
   `card-counting-stats`, `true-count-stats`, `deviation-stats`. The base
   `stats-store.ts` is tested, but each subclass wires a distinct
   `localStorage` key; the README emphasizes that each trainer must persist
   **independently** (and that mode switches must not clobber another mode's
   stats), so a small per-subclass test of key isolation is high-signal.
3. **Static chart data (`data/*.ts`)** — H17/S17 basic-strategy and
   deviation charts. These are transcribed from external PDFs; structural
   integrity tests (every expected row/column present, every cell a legal
   symbol) would guard against transcription drift without asserting
   specific strategy values.
4. **Presentational components** without specs (basic-strategy/*, shared/*,
   a few others) — lower priority and higher-effort (TestBed/DOM).

## 3. Highest-Value Coverage Gaps (to confirm, then implement)

### Gap 1 — `card-generator.service.ts` has no tests
- **Why it matters**: feeds every drill; RNG-driven generation is bug-prone
  at the edges (counts of 0/1, value ranges, sequence length, reproducibility
  with an injected RNG/seed if supported).
- **Suggested tests (confirm API first)**: a generated sequence has the
  requested length; every produced card has a valid rank/suit; with a fixed
  RNG the output is deterministic; boundary counts behave.
- **Risk**: Low. **Validation**: `npm test`.

### Gap 2 — `StatsStore` subclasses' key isolation
- **Why it matters**: README requires per-trainer/per-mode stats isolation;
  a wrong/shared `localStorage` key is a silent data-mixing bug.
- **Suggested tests**: each subclass reads/writes under its own documented
  key (see README's key table); resetting one does not affect another.
- **Risk**: Low. **Validation**: `npm test`.

### Gap 3 — Chart data integrity (`data/*.ts`)
- **Why it matters**: hand-transcribed from PDFs; structural drift is easy
  to introduce and hard to spot.
- **Suggested tests**: each chart exposes the full set of expected
  player-hand × dealer-upcard cells; every cell value is one of the legal
  encoded symbols documented in the README.
- **Risk**: Low. **Validation**: `npm test`.

### Gap 4 — Untested feature/shared components
- **Why it matters**: user-visible behavior (e.g. `card-stream`,
  `deviation-feedback-panel`, basic-strategy buttons/table).
- **Risk**: Medium/higher effort (TestBed). Deferred.
