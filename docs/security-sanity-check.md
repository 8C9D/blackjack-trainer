## 1. Scope

Practical security hygiene sanity check of the `blackjack-trainer` repository
on branch `chore/repo-cleanup-autopilot`. This is **not** a penetration test
or a full audit. It covers: tracked secrets/sensitive files, `.gitignore`

Inspection was local-only; no external services were contacted beyond

- **Deploy**: produces a static bundle in `dist/` (`ng build`).

## 3. Executive Summary

**Overall risk: Low.**

This is a static, frontend-only training app with no backend, no network
calls, no secrets, no authentication, and no user-supplied data leaving the
browser. `npm audit` reports **0 vulnerabilities**. The only hygiene gap was
that `.gitignore` did not protect `.env*` files; this has been fixed
defensively (no `.env` exists today, but the guard prevents a future
accidental commit). No real secrets were found. No manual security action is
required.

## 4. Findings

### Finding 1 — `.gitignore` did not ignore `.env*` files
- **Severity**: Low
- **Location**: `.gitignore`
- **Evidence**: `grep -E '^\.env' .gitignore` returned no matches before the
  fix; the file ignored `node_modules`, `dist`, IDE files, and OS files, but
  had no environment-file rule.
- **Why it matters**: If a contributor later introduces a `.env` with API
  keys or other secrets (common when a backend is added), the absence of an
  ignore rule makes accidental commits likely. Publish-safety hygiene.
- **Recommended fix**: Add `.env`, `.env.*`, and `!.env.example` to
  `.gitignore`.
- **Auto-fix status**: **Fixed** (see §11).
- **Secret redacted**: No (no secret involved).
- **Confidence**: High.

### Finding 2 — No tracked secrets or sensitive files
- **Severity**: Info
- **Location**: whole repo (150 tracked files).
- **Evidence**: No `.env*`, key, cert, keystore, dump, or `service-account`
  files are tracked or present on disk. A keyword scan
  (`API_KEY|SECRET|PASSWORD|PRIVATE_KEY|DATABASE_URL|BEGIN RSA|BEGIN OPENSSH`
  plus AWS/GitHub/Slack token shapes) produced only **two false positives**:
  (a) `public/cards/COPYING.txt:340` — the LGPL license body containing the
  phrase "password or key"; (b) a dependency name string in
  `package-lock.json`. Neither is a credential.
- **Why it matters**: Confirms nothing sensitive is committed.
- **Auto-fix status**: N/A (nothing to fix).
- **Secret redacted**: No real secret found; nothing to redact.
- **Confidence**: High.

### Finding 3 — No unsafe DOM / HTML rendering (XSS surface)
- **Severity**: Info
- **Location**: `src/`
- **Evidence**: Zero occurrences of `innerHTML`, `outerHTML`,
  `insertAdjacentHTML`, `document.write`, `eval(`, `new Function`, or
  Angular's `bypassSecurityTrust*` / `DomSanitizer`. Angular's default
  contextual auto-escaping is therefore intact and unbypassed.
- **Why it matters**: No injection path from data into the DOM.
- **Auto-fix status**: N/A.
- **Confidence**: High.

### Finding 4 — `localStorage` parsing is type-guarded
- **Severity**: Info
- **Location**: `src/app/core/services/stats-store.ts`
- **Evidence**: `load()` wraps `JSON.parse` in `try/catch` and only accepts
  the value after verifying all four fields are `number`s, copying them
  explicitly into a fresh object (no spread of untrusted keys). Malformed or
  missing payloads fall back to empty stats. `persist()` tolerates
  quota/private-mode throws.
- **Why it matters**: No prototype-pollution or crash surface from tampered
  `localStorage`; worst case is a stats reset, which is acceptable.
- **Auto-fix status**: N/A (already safe).
- **Confidence**: High.

### Finding 5 — Dependencies report no known vulnerabilities
- **Severity**: Info
- **Location**: `package.json` / `package-lock.json`
- **Evidence**: `npm audit --audit-level=moderate` → `found 0
  vulnerabilities`. `package.json` scripts are plain Angular CLI wrappers
  (`ng`, `ng serve`, `ng build`, `ng test`) — no `postinstall`/`preinstall`
  or other lifecycle scripts that run arbitrary code on install.
- **Auto-fix status**: N/A.
- **Confidence**: High.

## 11. Auto-Fixes Applied

### Fix 1 — Add `.env*` protection to `.gitignore`
- **Files changed**: `.gitignore`
- **What changed**: Added a section:
  ```gitignore
  # Environment variables / local secrets
  .env
  .env.*
  !.env.example
  ```
- **Why it is safe**: `.gitignore` affects only which untracked files Git
  surfaces; it changes no application code, build output, or runtime
  behavior. No `.env` file exists today, so nothing currently tracked is
  affected. The `!.env.example` negation preserves the ability to commit a
  documented example file later.
- **Validation run**: Created throwaway `.env` and `.env.example`;
  `git check-ignore .env` → `.env` (ignored), `git check-ignore .env.example`
  → not ignored (still trackable); `git status --porcelain` showed only
  `.gitignore` modified (confirming `.env` is hidden). Throwaway files
  removed. `git diff --check` → clean (no whitespace errors).
- **Commit hash**: see git log for the `chore: improve security hygiene`
  commit on this branch.
- **Push result**: pushed to `origin/chore/repo-cleanup-autopilot`.

No other auto-fixes were applied — no other safe, behavior-preserving
